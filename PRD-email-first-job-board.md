# 求职申请管理看板 PRD v2（Email-First / 单工作台交互版）

## 1. Summary
本 PRD 定义一个“从收到邮箱开始”的求职管理产品。产品不做投递记录，只做后续流程：测评、面试、补材料、结果跟进、复盘成长。  
本版重点是交互重构：`单工作台 + 弹窗/抽屉`，并明确“未登录可浏览，点击功能入口才要求登录”。

## 3. Background
### 3.1 Context
用户核心痛点不是“记得投没投”，而是“邮箱里后续流程太多，容易漏关键节点”。  
用户最常见失败点：
- 测评截止漏掉
- 面试准备不足
- 面试后不复盘，问题重复出现

### 3.2 Why This Version
上一轮方案功能完整，但页面概念偏多，用户对“如何一步步操作”不够确定。  
本版改为一个主工作台，把路径收敛成：
1. 看最近事项
2. 点功能入口
3. 进入弹窗处理
4. 回写流程卡

### 3.3 Design Principle
- 单页优先，减少跳转成本
- 动作优先，弱化复杂导航
- 先可浏览，再在关键操作点要求登录

## 4. Objective
### 4.1 Product Objective
让用户每天打开一次页面，就能清楚知道：
- 今天最先处理哪条测评/面试
- 每条流程下一步是什么
- 面试后如何沉淀改进动作

### 4.2 Business Objective
建立一个高频、低摩擦的求职流程入口，形成“准备-面试-复盘”闭环使用习惯。

### 4.3 Key Results（SMART）
**O1：用户可在单工作台内完成邮箱后流程管理。**
- KR1：首次访问时“连接邮箱弹窗”触达率 = 100%（M1）。
- KR2：未登录状态下点击功能入口触发登录引导成功率 >= 99%（M1）。
- KR3：连接邮箱后 30 秒内完成初次列表渲染成功率 >= 95%（M2）。
- KR4：首屏“第一个流程项目默认展开”命中率 = 100%（M1）。
- KR5：每周至少上传 1 次复盘材料的活跃用户占比 >= 40%（M3）。

## 5. Market Segment(s)
### Segment A：并行求职执行者（核心）
- 同时推进多家公司流程
- 需要“按时间优先处理”而不是“全量记录”

### Segment B：面试密集准备者
- 1-2 周内多场面试
- 需要一面/二面分轮准备入口和复盘沉淀

### Segment C：复盘驱动提升者
- 关注“哪类问题反复出现”
- 需要跨场次短板聚合和下次行动建议

## 6. Value Proposition(s)
### 6.1 Jobs To Be Done
- 把邮箱事件变成可执行动作
- 用最短路径推进流程，不漏关键节点
- 面试后把经验沉淀成下次可用清单

### 6.2 Gains
- 单页即可看总情况、筛选、最近事项、流程处理
- 关键动作入口固定：`流程 / JD识别 / 一面 / 二面 / 复盘上传`
- 复盘可回写到流程卡，形成闭环

### 6.3 Pain Relievers
- 不再在多个页面来回切换
- 不登录也可先浏览内容，不会“被锁死”
- 真正操作时才要求登录，降低首屏阻力

### 6.4 Against Alternatives
- 比 Excel/Notion 少手工维护
- 比普通待办多求职语义（轮次、测评、复盘）
- 比纯面经工具多“流程推进 + 时间优先”

## 7. Solution
### 7.1 UX / Prototypes（单工作台）
#### 路由级
1. `/workspace`（唯一业务页面）
2. `/auth/callback`（技术回调）

#### 页面分区（自上而下）
1. 顶栏：邮箱状态 + 连接/退出
2. 总情况：待处理邮件、测评待完成、最近面试、逾期风险
3. 筛选条：关键词、阶段、风险
4. 最近事项区：默认展开（最近面试/测评优先）
5. 邮件解析队列：事件确认入流程
6. 流程处理区：项目卡列表（首条默认展开）

#### 流程卡展开区（强约束）
每条展开卡必须含 5 个动作入口：
1. `流程`（打开详情抽屉）
2. `JD识别`（弹窗）
3. `一面`（icon）
4. `二面`（icon）
5. `复盘上传`（弹窗）

### 7.2 Key Features（含交互验收）
#### F1. 登录引导与软拦截
- 首次进入自动弹连接邮箱弹窗
- 用户关闭后可继续浏览和滚动
- 未登录点击任意功能入口时，弹登录引导

**验收标准**
1. 未登录可浏览列表，不可直接执行功能动作
2. 点击功能入口（流程/JD/一面/二面/复盘）必弹登录引导

#### F2. 连接后首次加载
- 授权成功后显示加载态（Skeleton）
- 加载完成自动展示最近事项和流程区

**验收标准**
1. 有可见加载态
2. 加载后统计卡与列表数据一致

#### F3. 最近事项区（时间优先）
- 按最近时间排序
- 展示测评与面试节点
- 支持折叠/展开

#### F4. 流程处理区（首条默认展开）
- 展示项目列表（公司、岗位、节点、时间、风险）
- 第一个项目默认展开
- 点击其他项目可切换展开

**验收标准**
1. 页面初渲染时第一个项目展开
2. 展开区总是只有一个当前项

#### F5. 邮件解析队列
- 邮件事件候选展示：公司、岗位、事件类型、截止时间、置信度
- 支持“确认并入流程”

#### F6. 机会详情抽屉（流程）
- 显示时间线、下一步动作、快捷入口

#### F7. JD 上传识别
- 支持上传文件或粘贴文本
- 返回结构化摘要

#### F8. 一面/二面准备
- 用户点击 icon 才打开对应轮次准备面板
- 不主动打断用户主流程

#### F9. 复盘上传
- 支持录音或文档上传
- 生成后回写该流程卡的下一步动作与时间线

### 7.3 Technology（高层）
- 前端：单页状态驱动（Auth State + UI State + Pending Action）
- 后端：邮箱事件抽取服务 + 复盘生成服务
- 数据：Opportunity、EmailEvent、InterviewSession、ReviewDoc
- 安全：最小权限邮箱读取、断连与数据删除

### 7.4 Assumptions（待验证）
1. 用户接受“先浏览后登录”的门槛设计
2. 功能入口拦截不会引发明显流失
3. 一面/二面 icon 入口比独立页面更高效
4. 复盘上传后回写能提升下次准备完成率

## 8. Release
### 8.1 Scope Boundary（V1 强约束）
**明确不做**
- 投递记录管理
- 职位推荐/职位搜索
- 社区互动功能

### 8.2 Milestones（相对周期）
#### M1（1-2 周）：交互框架可用
- 单工作台结构
- 登录引导与拦截状态机
- 首条默认展开

#### M2（2-4 周）：流程主链路打通
- 邮件解析候选 -> 入流程
- 详情抽屉 + JD识别 + 一/二面准备入口

#### M3（2-3 周）：复盘闭环
- 复盘上传
- 结果回写流程卡
- 基础改进建议展示

### 8.3 MVP Go-Live Gate
上线前必须满足：
1. 未登录可浏览，功能点击可正确触发登录引导
2. 连接邮箱后可稳定进入加载 -> 可操作状态
3. 第一个流程项目默认展开且包含 5 个动作入口
4. 复盘上传后能回写时间线和下一步动作

### 8.4 Risks & Mitigations
- 风险：用户对邮箱权限敏感  
  缓解：清晰权限文案、可随时断连  
- 风险：事件抽取不稳定  
  缓解：展示置信度 + 人工确认入口  
- 风险：复盘使用率低  
  缓解：先支持文档上传，逐步引导录音上传
